# Odoo `addons/web` 模块评价

> **版本**: 1.0  
> **日期**: 2026-05-31  
> **主题**: 评价 Odoo 19 CE `addons/web` 模块在 Odoo 原生框架中的作用、定位与质量

---

## 一、模块定位

### 1.1 基础信息

| 属性 | 值 | 含义 |
|------|-----|------|
| 模块名 | `web` | Odoo Web Core |
| 依赖 | `['base']` | 仅依赖最低层核心框架 |
| 自动安装 | `True` | 不可卸载，任何依赖它时自动安装 |
| 被依赖 | 37+ 个 addon | `sale`, `crm`, `stock`, `account`, `hr`... 所有业务模块 |
| 服务端 | 6,498 行 Python (33 files) | ORM 桥接 + HTTP/JSON-RPC 控制器 |
| 前端 | 114,204 行 JS + 14,068 行 SCSS | Owl SPA 框架 + 视图引擎 + Widget 系统 |
| 测试 | 3,412 行 Python + 375 JS 测试 | — |
| **总计** | **~138,000 行** | — |

### 1.2 架构中的位置

`web` 模块是 Odoo 架构中 **唯一的前端基础设施层**，处于应用层与核心框架之间：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  应用层 (业务模块)
  sale / crm / stock / account / hr / ...
  全部依赖 "web" → 通过 ir.ui.view + ir.actions 声明 UI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
          │
          ▼
  ┌───────────────────────────────────────┐
  │          addons/web                   │
  │                                        │
  │  服务端 (5%):                          │
  │    models/models.py       2,294 lines  │  ORM-web 桥接
  │    models/ir_http.py        205 lines  │  会话工厂
  │    controllers/             576 lines  │  HTTP/JSON-RPC API
  │    models/others             ~400 lines│  模型扩展
  │                                        │
  │  客户端 (95%):                         │
  │    Owl 组件框架                       │  SPA 运行时
  │    7 种视图引擎                       │  Form/List/Kanban/...
  │    87 种字段 Widget                   │  类型→组件映射
  │    Action/Menu/Session 管理           │  导航与状态
  │    RPC 客户端 + WebSocket             │  通信层
  └───────────────────────────────────────┘
          │
          ▼
  ┌───────────────────────────────────────┐
  │          addons/base                   │
  │  ORM 引擎 / 元数据 (ir.*) / 工作流     │
  └───────────────────────────────────────┘
```

## 二、职责划分

### 2.1 服务端职责

`web` 模块的服务端只占 **5%** 代码量（6,498 行），但承载 **100%** 的业务数据供给责任：

| 组件 | 文件 | 行 | 职责 |
|------|------|:--:|------|
| ORM 桥接 | `models/models.py` | 2,294 | `web_read` / `web_search_read` / `web_read_group` — 所有前端数据查询的统一入口 |
| 会话管理 | `models/ir_http.py` | 205 | `session_info()` — 返回 50+ 字段的用户/系统状态 |
| API 路由 | `controllers/` (7 个核心) | 576 | 20+ HTTP/JSON-RPC 端点 |
| 文件服务 | `controllers/binary.py` | 333 | 图片、附件、文档的流式传输 |
| 数据导出 | `controllers/export.py` | 713 | CSV/XLSX 格式转换 |

**核心设计**: 服务端代码极少，但每个方法都是经过多年验证的 **稳定 API 契约**。所有业务模块的 UI 数据都通过这 2,294 行的 `models.py` 提供。

### 2.2 客户端职责

`web` 模块的客户端占 **95%** 代码量（~128,000 行），负责：

| 组件 | 职责 |
|------|------|
| Owl SPA 框架 | 响应式组件系统、虚拟 DOM、模板引擎 |
| 视图引擎 | 解析 XML `arch` → 渲染 7 种视图类型 |
| Widget 系统 | 87 种字段类型的编辑/只读组件 |
| Action 系统 | 6 种动作类型的解析与执行 |
| 菜单系统 | `load_menus` → 导航栏 + App 切换器 |
| RPC 客户端 | 封装 `call_kw` / `web_search_read` 等远程调用 |
| WebSocket | Odoo Bus 事件实时通知 |

### 2.3 不负责的领域

`web` 模块 **不是**：

| 不是 | 说明 |
|------|------|
| 业务逻辑引擎 | 销售流程、会计规则等由业务模块定义 |
| 权限系统 | 访问控制由 `ir.model.access` + `ir.rule` 实现 |
| 数据库层 | 存储和查询由 PostgreSQL + ORM 引擎完成 |
| 报表模板 | PDF 排版由 `qweb-pdf` 类型模板定义 |
| 工作流引擎 | 自动化由 `base.automation` + `ir.cron` 驱动 |

## 三、设计哲学

### 3.1 瘦服务端-胖客户端

```
Odoo 特有架构:

  服务端 (5% 代码)          客户端 (95% 代码)
  ──────────────            ─────────────────
  少量关键 API              大量视图/交互代码
  稳定契约 (API 不变)        可替换 (OWL → React)
  单点数据供给              多态 UI 呈现
```

**Odoo 的独特之处**: web 模块实现了服务端-客户端的极端不对称分工。服务端用极少代码提供统一数据接口，客户端用大量代码实现丰富的 UI。这使得：

1. **模块化成为可能**: 任何业务模块只需定义 XML 视图 + Python 模型，无需写前端代码
2. **前端可替换**: 遵守 `web_search_read` / `get_views` 的 API 契约，即可用任意技术栈重建 UI
3. **业务与 UI 彻底解耦**: 业务逻辑变更不影响前端，前端框架升级不影响业务

### 3.2 通用基础设施

`web` 模块的关键设计决策：**不包含任何业务逻辑**。

- 不硬编码 "销售订单" 的字段列表 — 业务模块通过 `ir.ui.view` 声明
- 不预设 "CRM 管道" 的看板卡片 — 业务模块通过 `<kanban>` XML 定义模板
- 不固话任何菜单结构 — 运行时从 `ir.ui.menu` 动态加载

这使得 Odoo 成为一个 **通用 ERP 框架** 而非特定领域的应用。

### 3.3 声明式元数据驱动

```
业务模块只需声明:

  1. 模型定义      → ir.model / ir.model.fields
  2. 视图布局      → ir.ui.view (XML)
  3. 菜单入口      → ir.ui.menu
  4. 动作定义      → ir.actions.*
  5. 权限规则      → ir.rule

web 模块自动:
  - 解析 XML → 渲染 UI
  - 加载菜单 → 构建导航
  - 解释规则 → 过滤数据
  - 处理 CRUD → 通过 ORM 桥接
```

| 维度 | 评分 | 评语 |
|------|:----:|------|
| **架构设计** | 9/10 | 瘦服务端-胖客户端分离是经典设计，声明式元数据驱动是核心竞争力 |

## 六、对 oweb 的启示

### 6.1 继承策略

```
Odoo addons/web                 oweb 处理方式
─────────────────                ───────────────
服务端 (5% = 6,498 行)    →    ✅ 完全继承，通过 BFF 透明代理
客户端 (95% = 128,000 行)  →    ❌ 完全替换为 React (apps/oweb)
QWeb 模板引擎              →    ⚠️ 语义映射，不移植
CSS/Bootstrap              →    ❌ 替换为 Tailwind
```

### 6.2 关键洞察

1. **服务端不可动**: `models.py` 2,294 行是 Odoo 的核心资产，oweb 只需代理调用
2. **客户端可替换**: Owl 117,000+ 行正好是 oweb 要替换的目标
3. **API 契约是边界**: `web_search_read`/`get_views`/`session_info` 的格式是稳定接口
4. **95% 是前端代码**: web 模块本质是"超大前端框架 + 极瘦 API 层"
5. **声明式元数据是灵魂**: oweb 完整保留了 `ir.ui.view` XML → 视图 的驱动链

### 6.3 oweb 的架构优势

| Odoo 原生 | oweb | 优势 |
|-----------|------|------|
| Owl (自研框架) | React 19 | 全球最大的前端生态 |
| jQuery 遗留代码 | 纯 React | 零遗留依赖 |
| Bootstrap 5 | Tailwind CSS 4 | 现代 Utility-first |
| 无 HMR | Vite HMR | 秒级热更新 |
| require.js | Vite Code Splitting | Tree Shaking |
| OWL 学习成本 | React 标准 | 人才池广 |

## 七、总结

`addons/web` 在 Odoo 原生框架中的角色是 **唯一的通用 UI 基础设施**。它实现了：

- **服务端**: 极简但完备的数据 API 层（占模块的 5%）
- **客户端**: 功能完整的 SPA 框架（占模块的 95%）
- **核心价值**: 声明式元数据驱动的 UI 渲染——业务模块只需声明模型和视图，无需写前端代码

oweb 项目继承其服务端 API 契约（通过 BFF 代理），替换其客户端实现（用 React），完整保留了 Odoo 的元数据驱动架构哲学。

**服务端不可动，客户端尽可替。** 这是 oweb 存在的根本理由。
